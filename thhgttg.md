# pathogen

**pathogen** is a framework for building systems where meaning comes first and structure emerges from action.

Most systems start with structure — schemas, hierarchies, permission tables — and then try to layer meaning on top. This gets it backwards. Structure that arrives before anyone knows why it's there becomes the thing everyone works around instead of with.

pathogen inverts this: start with intent, let structure resolve from interaction, make trust explicit, keep everything legible and reversible.

The framework applies at every scale — from replacing a single server to rethinking how political systems handle conflict.

---

## The Core Idea

All of pathogen rests on one principle:

> **Differences are primary. Values are derived.**

Intent decomposes into change. Change composes under constraints. Stable structure is what settles out. This is how physical systems work. pathogen applies the same logic to computation, organizations, and policy.

- **[Field guide](delete4.md.)** — The operational manual. How to manage conflict as a control problem. Don't oppose the pull — place a better attractor. Immediately actionable without needing to understand the rest of the framework.

## The Connection

These documents are all the same idea at different scales:

- **foldr** says: computation is difference resolution
- **bootstrap** says: systems should start from intent, not schema
- **nb** says: the file is the interface, not a container for data
- **license42** says: policy is routing, not authority
- **learning-phase** says: persistent problems persist because the frame is wrong
- **high-gain** says: in hostile environments, seeds survive by not looking like seeds
- **field-guide** says: place competing attractors, keep them boring, reward what persists

The underlying claim is: **free will only means something if you use it to connect.** Independence without interdependence produces fragmentation. Structure without meaning produces bureaucracy. Meaning without structure dissolves.

The goal is freely chosen interdependence — systems where trust is explicit, intent is legible, and the infrastructure serves connection rather than commanding it.

## Design Principles

- Differences are primary; values are derived
- Intent is explicit and inspectable
- Constraints are first-class
- Composition beats sequencing
- Stability is designed, not assumed
- Control should be legible and reversible
- Make it understandable before it is powerful
